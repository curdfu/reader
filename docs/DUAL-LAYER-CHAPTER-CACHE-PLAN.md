# 双层章节缓存实施计划

## 目标

在不改变现有缓存入口的前提下，将“缓存书籍”升级为双层缓存：

1. 服务端保存一份持久章节缓存。
2. 当前阅读端保存一份 `window.$cacheStorage` 缓存。
3. 后续阅读章节时优先读取阅读端缓存。
4. 阅读端缓存未命中时再请求服务端。
5. 服务端优先读取本地持久缓存，未命中时再访问书源。

目标读取链路：

```text
阅读章节
  -> 阅读端 window.$cacheStorage
      -> 命中：直接显示
      -> 未命中：请求服务端 /reader3/getBookContent
          -> 服务端本地章节缓存命中：返回缓存
          -> 服务端本地章节缓存未命中：访问书源
      -> 前端收到成功响应后写入 window.$cacheStorage
```

章节内容需要能显示本次实际来源：

```text
阅读端本地缓存
服务器缓存
书源
```

缓存入口目标链路：

```text
用户点击现有缓存入口
  -> 服务端批量缓存章节
  -> 前端同步或随后从服务端拉取已缓存章节
  -> 写入当前阅读端 window.$cacheStorage
```

## 现有基础

当前工程已经有相关基础能力：

- `BookController.cacheBookSSE`：服务端批量缓存章节。
- `BookController.getBookContent`：获取章节内容，并已有服务端章节缓存相关逻辑。
- `BookController.getShelfBookWithCacheInfo`：返回书籍缓存状态。
- `BookController.deleteBookCache`：删除服务端章节缓存。
- `Reader.vue`：阅读页已有“后面 50 章 / 后面 100 章 / 后面全部”的缓存入口。
- `BookManage.vue`：书籍管理页已有“缓存到服务器”“缓存到浏览器”“删除浏览器缓存”等逻辑。
- `helper.js`：`networkFirstRequest` 和 `cacheFirstRequest` 已支持 `window.$cacheStorage`。

现有入口语义需要在实现时统一：

- “缓存到服务器”：升级为双层缓存入口，即服务端持久缓存 + 当前阅读端缓存补齐。
- “缓存到浏览器”：可保留为辅助入口，仅补齐当前阅读端缓存，不删除、不替代服务端缓存。
- 阅读页“后面 50 章 / 后面 100 章 / 后面全部”：保持阅读端预缓存语义，成功响应仍写入 `window.$cacheStorage`；是否同步触发服务端持久缓存由 `getBookContent` 的 `cache` 参数和后端策略决定。

现有阅读端缓存 key 形态：

```text
localCache@{book.name}_{book.author}@{book.bookUrl}@chapterContent-{index}
```

这个 key 已经包含 `bookUrl`，不同书源或换源后的同名书不会天然冲突。

## 非目标

本计划不要求浏览器缓存永久可靠。

允许以下情况导致阅读端缓存丢失：

- 用户清理浏览器缓存或站点数据。
- 浏览器在存储压力下清理站点数据。
- 隐私模式或受限 WebView 环境。
- 不同域名、端口、协议之间不共享缓存。
- iOS / iPadOS / macOS Safari 根据系统策略清理站点数据。
- Android WebView 或系统浏览器在存储压力下清理站点数据。

例如以下访问入口互不共享阅读端缓存：

```text
https://reader.curdfu.top:18940
http://localhost:18910
```

## 阅读端存储兼容性要求

阅读端缓存需要同时考虑以下环境：

- Android Chrome / Android WebView / Android 系统浏览器。
- iOS Safari。
- iPadOS Safari。
- iOS / iPadOS 上的第三方浏览器。
- macOS Safari。
- macOS Chrome / Edge / Firefox。

### 推荐存储优先级

推荐继续以 `window.$cacheStorage` 作为章节正文缓存入口，但实现和验证时需要确认它的底层能力。

建议优先级：

```text
IndexedDB / localForage 类异步存储
  -> 可用：作为章节正文主缓存
  -> 不可用：降级为 localStorage，只缓存少量章节或禁用阅读端缓存
```

不建议用 `localStorage` 作为正文主缓存：

- 容量通常较小。
- 同步 API，章节多时容易卡 UI。
- 移动端更容易触发配额异常。

不建议只依赖 Service Worker Cache Storage：

- 当前章节内容是业务 JSON，不是纯静态资源。
- 本工程已有 `window.$cacheStorage` 路径。
- localhost 开发入口当前会注销 Service Worker。

### 平台差异

#### Android

Android Chrome 通常对 IndexedDB 支持较好，适合作为主要验证环境。

风险：

- Android WebView 的存储策略可能由外层 App 控制。
- 厂商浏览器可能更激进地清理站点数据。
- 存储空间不足时，浏览器可能清理非持久站点数据。

验证重点：

- 批量缓存 100 章后刷新页面仍存在。
- 关闭浏览器再打开仍存在。
- Android WebView 环境下不会因为 IndexedDB 不可用直接报错。

#### iOS / iPadOS

iOS 和 iPadOS 上所有浏览器都受 WebKit 存储策略影响，即使是 Chrome / Edge 也不是 Chromium 内核。

风险：

- Safari 可能按系统策略清理站点数据。
- 普通网页和添加到主屏幕的 PWA，在存储表现上可能不同。
- 低存储空间、长时间不访问、隐私模式等场景下缓存可能丢失。
- IndexedDB 可用但配额和清理策略不可完全预测。

验证重点：

- Safari 普通标签页。
- 添加到主屏幕后打开。
- iPadOS 分屏或后台切换后缓存是否仍可读。
- 缓存大量章节时是否出现配额错误。

#### macOS

macOS Chrome / Edge / Firefox 通常对 IndexedDB 支持稳定。

macOS Safari 仍需要单独验证。

风险：

- Safari 的站点数据清理策略与 Chromium 不同。
- 用户开启隐私浏览时缓存不可依赖。

验证重点：

- macOS Safari。
- macOS Chrome。
- 刷新页面、关闭浏览器、重启浏览器后缓存读取。

### 存储能力探测

实现时建议在应用启动或首次缓存前做能力探测：

1. 检查 `window.$cacheStorage` 是否存在。
2. 写入一条小测试数据。
3. 读取测试数据。
4. 删除测试数据。
5. 如果失败：
   - 禁用阅读端批量缓存。
   - UI 提示当前浏览器不支持阅读端缓存。
   - 服务端缓存功能仍可使用。

可选增强：

```javascript
if (navigator.storage && navigator.storage.estimate) {
  // 估算 quota / usage，用于提示剩余空间风险
}
```

不建议把 `navigator.storage.persist()` 作为强依赖：

- Safari 支持有限。
- 用户授权和浏览器策略不可控。
- 本需求本身不要求稳定永久保存。

### 配额和降级策略

章节缓存写入时必须处理配额异常。

建议行为：

1. 单章写入失败：
   - 不影响阅读。
   - 记录该章阅读端缓存失败。
2. 连续多章失败：
   - 停止阅读端补齐。
   - 服务端缓存继续。
   - UI 提示“当前浏览器存储空间不足，已停止缓存到阅读端”。
3. 阅读端缓存不可用：
   - 继续保留服务端缓存能力。
   - 阅读时走服务端和书源。

### 跨端验收要求

基础验收至少覆盖：

| 平台 | 浏览器 | 必测项 |
| --- | --- | --- |
| Android | Chrome | 批量缓存、刷新后读取、关闭后读取 |
| Android | WebView 或系统浏览器 | 存储能力探测、失败降级 |
| iOS | Safari | 批量缓存、刷新后读取、后台恢复 |
| iPadOS | Safari | 批量缓存、分屏/后台恢复 |
| iOS/iPadOS | 第三方浏览器 | 确认 WebKit 下缓存行为 |
| macOS | Safari | IndexedDB 可用性和刷新后读取 |
| macOS | Chrome/Edge/Firefox | 批量缓存和关闭后读取 |

验收结论只承诺：

```text
在浏览器未清理站点数据的情况下，阅读端缓存应尽量保持可读。
```

不承诺：

```text
跨浏览器共享。
跨设备共享。
跨域名/端口共享。
系统清理后恢复。
```

## 阶段 2：阅读链路改成阅读端优先

### 目标

后续阅读章节时，优先从 `window.$cacheStorage` 读取。

来源字段取值在实现中统一为：

```text
localCache    阅读端本地缓存
serverCache   服务器缓存
source        书源
```

### 计划

1. 在 `Reader.vue` 的章节加载链路中加入阅读端 cache-first 逻辑。
2. 读取章节前先构造当前章节的阅读端缓存 key。
3. 如果 `refresh !== true` 且阅读端缓存命中：
   - 直接使用缓存内容。
   - 保持现有显示流程。
   - 继续调用现有 `addChapterContentToCache`，保持内存缓存一致。
   - 触发分页、滚动、多章节渲染等现有后续流程。
4. 如果阅读端缓存未命中：
   - 调用服务端 `/reader3/getBookContent`。
   - 请求成功后写入 `window.$cacheStorage`。
5. 如果 `refresh === true`：
   - 跳过阅读端缓存。
   - 请求服务端刷新。
   - 成功后覆盖阅读端缓存。

### 注意点

- 不要让错误内容覆盖正确缓存。
- 如果请求失败，可以再尝试读取阅读端缓存兜底。
- 滚动阅读会同时加载前后多章，需要复用同一套 cache-first 逻辑。
- EPUB、CBZ、音频等特殊内容要谨慎处理，可以先仅保证文字章节。

### 验收

1. 阅读端已缓存章节：
   - 不请求服务端即可显示。
   - 来源显示为“本地缓存”或“阅读端缓存”。
2. 阅读端未缓存章节：
   - 正常请求服务端。
   - 成功后写入阅读端缓存。
   - 如果服务端返回缓存内容，来源显示为“服务器缓存”。
   - 如果服务端访问书源，来源显示为“书源”。
3. 强制刷新：
   - 跳过阅读端缓存。
   - 成功后覆盖阅读端缓存。
   - 来源不应显示为“本地缓存”。
   - 如果服务端访问书源，来源显示为“书源”。
   - 如果后端实现允许刷新后重新写入并返回服务器缓存，也必须明确标记真实来源，不能误标。
4. 服务端不可用但阅读端缓存存在：
   - 已缓存章节仍可显示。
   - 来源显示为“本地缓存”或“阅读端缓存”。

## 阶段 3：服务端读取优先本地持久缓存

### 目标

当前端未命中阅读端缓存并请求服务端时，服务端优先返回本地章节缓存。

### 计划

1. 检查 `BookController.getBookContent` 当前是否只在 `cache=1` 或特定配置下读取服务端缓存。
2. 调整为默认优先读取服务端本地章节缓存。
3. `refresh > 0` 时跳过服务端本地缓存，重新访问书源。
4. 服务端访问书源成功后，是否自动写入服务端缓存需要明确：
   - 用户点击缓存入口时必须写入。
   - 普通阅读是否自动写入，可以保持现状，避免服务端缓存无限增长。

推荐策略：

```text
普通阅读：
  - 优先读服务端已有缓存
  - 未命中则访问书源
  - 是否写服务端缓存保持现有配置

缓存入口触发：
  - 必须写服务端缓存
```

### 验收

1. 服务端已有缓存时：
   - `/getBookContent` 直接返回缓存内容。
   - 响应中能标记来源为 `serverCache`。
2. 服务端没有缓存时：
   - 正常访问书源。
   - 响应中能标记来源为 `source`。
3. `refresh > 0`：
   - 强制绕过服务端缓存。
   - 响应来源不应误标为 `serverCache`。
4. 服务端缓存文件不因普通阅读失控增长。

## 阶段 3.5：章节来源标识

### 目标

用户能在阅读界面看到当前章节内容来自哪里。

来源分三类：

| 来源 | 含义 |
| --- | --- |
| 阅读端本地缓存 | 命中当前浏览器的 `window.$cacheStorage` |
| 服务器缓存 | 前端请求服务端，服务端命中本地章节缓存 |
| 书源 | 服务端访问原始书源后返回 |

### 数据结构建议

当前 `/getBookContent` 返回值是章节正文字符串。

为了兼容现有调用，建议采用渐进式改造：

#### 方案 A：返回结构化对象

```json
{
  "content": "章节正文",
  "contentSource": "serverCache"
}
```

可选来源值：

```text
localCache
serverCache
source
```

优点：

- 语义清晰。
- 后续可扩展缓存时间、章节 hash、缓存命中耗时等信息。

风险：

- 需要适配所有当前假设 `res.data.data` 是字符串的前端代码。
- EPUB、图片、音频等特殊返回也要兼容。

#### 方案 B：保持正文字符串，额外响应字段标记来源

```json
{
  "isSuccess": true,
  "data": "章节正文",
  "meta": {
    "contentSource": "serverCache"
  }
}
```

优点：

- 对现有前端正文读取影响较小。
- 更适合渐进改造。

风险：

- 需要确认 `ReturnData` 是否适合携带 `meta`。

#### 推荐

优先选方案 B。

如果 `ReturnData` 不适合加 `meta`，可以在 `data` 外新增兼容字段，例如：

```json
{
  "isSuccess": true,
  "data": "章节正文",
  "contentSource": "serverCache"
}
```

### 前端处理

前端需要维护当前章节来源状态：

```text
currentChapterContentSource
```

阅读端命中 `window.$cacheStorage` 时：

```text
currentChapterContentSource = "localCache"
```

请求服务端成功时：

```text
currentChapterContentSource = res.data.contentSource || res.data.meta?.contentSource || "source"
```

如果服务端暂时还没有返回来源字段，则默认：

```text
source
```

### 展示位置建议

不建议放在正文中，避免干扰阅读。

推荐位置：

1. 阅读页顶部或底部状态栏，显示一个轻量标签：

```text
本地缓存
服务器缓存
书源
```

2. 仅在唤出菜单或底栏显示，不常驻正文区域。
3. 如果当前 UI 已有章节标题、进度、时间区域，优先放在同一状态区域。

移动端显示建议：

- 标签尽量短：
  - 本地
  - 服务器
  - 书源
- 或使用图标 + tooltip / title。

桌面端显示建议：

- 可以显示完整文字：
  - 阅读端缓存
  - 服务器缓存
  - 书源实时获取

### 多章节滚动模式

滚动阅读可能同时显示多章。

建议初期只显示“当前阅读章节”的来源，而不是每个章节都显示。

后续如果需要更精细，可以在每个章节标题旁显示来源：

```text
第 10 章 标题  [本地缓存]
```

但初期不建议这么做，避免 UI 变重。

### 验收

1. 阅读端缓存命中：
   - 页面不请求服务端。
   - 来源显示“本地缓存”。
2. 阅读端未命中、服务端缓存命中：
   - 页面请求服务端。
   - 服务端返回缓存内容。
   - 来源显示“服务器缓存”。
3. 双层缓存都未命中：
   - 服务端访问书源。
   - 来源显示“书源”。
4. 强制刷新：
   - 不显示“本地缓存”。
   - 如果服务端绕过本地缓存访问书源，则显示“书源”。
5. 服务端异常但本地缓存兜底：
   - 正文可显示。
   - 来源显示“本地缓存”。

## 阶段 4：统一缓存入口语义

### 目标

现有入口不变，但点击“缓存”后同时完成服务端缓存和阅读端缓存补齐。

### 计划

1. 保留 `BookManage.vue` 现有入口，不新增入口。
2. 将“缓存到服务器”升级为双层缓存入口。
3. 点击“缓存到服务器”时启动服务端 `cacheBookSSE`。
4. 服务端每缓存成功一章后，前端触发该章节的阅读端缓存补齐。
5. 阅读端补齐方式：
   - 调用 `getBookContent(index, options, false, true, book)`。
   - 服务端此时应优先从本地缓存返回。
   - 前端收到成功响应后写入 `window.$cacheStorage`。
6. 如果服务端 SSE 当前没有返回 `chapterIndex`，需要扩展 SSE 事件数据。
7. “缓存到浏览器”入口可以保留为仅补齐阅读端缓存的辅助操作，不能删除或覆盖服务端缓存。

建议 SSE 数据至少包含：

```json
{
  "chapterIndex": 12,
  "cachedCount": 13,
  "failedCount": 0,
  "totalCount": 100
}
```

### 页面关闭行为

如果用户关闭页面：

- 服务端已经完成的缓存保留。
- 阅读端未完成的缓存可以中断。
- 下次打开缓存管理页时，可以根据服务端缓存状态继续补齐阅读端缓存。

### 验收

1. 用户点击一次缓存入口：
   - 服务端缓存数量增加。
   - 浏览器缓存数量也增加。
2. 服务端已有缓存章节：
   - 阅读端补齐时不再访问书源。
3. 页面刷新后：
   - 浏览器缓存统计仍存在。
4. 页面关闭后：
   - 服务端已缓存内容仍存在。

## 阶段 5：缓存状态和统计

### 目标

让用户能区分服务端缓存和阅读端缓存。

### 计划

1. 保留现有两类统计：
   - 服务器缓存 N 章。
   - 浏览器缓存 N 章。
2. 服务端缓存中：
   - 由 SSE 更新服务器缓存数。
3. 阅读端补齐中：
   - 每完成一批章节后重新计算浏览器缓存数。
   - 避免每章都全量扫描，防止 UI 卡顿。
4. 缓存结束后统一刷新统计。

### 验收

1. 缓存过程中服务器缓存数递增。
2. 浏览器缓存数也逐步递增。
3. 缓存结束后两边数量合理。
4. 切换书籍管理页后统计仍正确。

## 阶段 6：删除和失效策略

### 删除服务端缓存

行为：

- 只删除服务端章节缓存。
- 不删除阅读端 `window.$cacheStorage`。

理由：

- 阅读端缓存是当前浏览器的弱离线副本。
- 删除服务端缓存不应该破坏用户当前设备已有阅读能力。

### 删除浏览器缓存

行为：

- 只删除 `window.$cacheStorage` 中该书章节内容。
- 不删除服务端缓存。

理由：

- 用户可能只是清理当前设备缓存。
- 服务端缓存仍可再次补齐阅读端。

### 删除书籍

建议：

- 保持现有书籍删除语义。
- 是否同步删除浏览器缓存需要单独确认。
- 如果实现，建议作为显式选项或默认清理该书阅读端缓存。

### 换源

当前 key 包含 `bookUrl`，换源后缓存天然隔离。

验收：

1. 删除服务端缓存后，阅读端已缓存章节仍可读。
2. 删除浏览器缓存后，服务端缓存仍可重新拉回。
3. 换源后不会误读旧源章节。

## 阶段 7：验证矩阵

### 场景 1：双层都没有缓存

前置：

- 阅读端无缓存。
- 服务端无缓存。

预期：

- 请求服务端。
- 服务端访问书源。
- 前端成功显示章节。
- 前端写入阅读端缓存。

### 场景 2：只有服务端缓存

前置：

- 阅读端无缓存。
- 服务端有缓存。

预期：

- 请求服务端。
- 服务端直接返回本地缓存。
- 前端显示章节。
- 前端写入阅读端缓存。

### 场景 3：只有阅读端缓存

前置：

- 阅读端有缓存。
- 服务端无缓存或不可用。

预期：

- 直接从阅读端缓存显示。
- 不请求服务端。

### 场景 4：双层都有缓存

前置：

- 阅读端有缓存。
- 服务端有缓存。

预期：

- 优先使用阅读端缓存。
- 不请求服务端。

### 场景 5：强制刷新

前置：

- 阅读端有缓存。
- 服务端有缓存。

预期：

- 跳过阅读端缓存。
- 服务端跳过或更新本地缓存。
- 成功后覆盖阅读端缓存。

### 场景 6：缓存入口双写

前置：

- 用户在书籍管理或阅读页点击现有缓存入口。

预期：

- 服务端缓存开始增长。
- 阅读端缓存随后增长。
- UI 显示两类缓存数量。

## 推荐实现顺序

1. 先实现阅读端 cache-first 读取。
2. 增加章节来源标识的前后端字段和显示。
3. 再调整服务端 `getBookContent` 的本地缓存优先策略。
4. 再将“缓存到服务器”入口升级为“服务端缓存 + 阅读端补齐”。
5. 最后优化统计、删除和异常提示。

理由：

- 阅读端 cache-first 是用户感知最明显的能力。
- 来源标识可以帮助验证 cache-first、server-cache、source 三条路径是否真的命中。
- 服务端缓存优先可以减少补齐阅读端缓存时的源站压力。
- 双写入口最后串联，风险最低。

## 风险

### 浏览器缓存不可靠

`window.$cacheStorage` 不是永久存储，可能被浏览器清理。

应在 UI 或文档中明确：

```text
浏览器缓存仅保存在当前设备和当前访问地址下，清理浏览器数据后会丢失。
```

其中 iOS、iPadOS、macOS Safari 需要重点验证，因为 WebKit 的站点数据清理策略更不可控。

### 访问地址隔离

不同 origin 不共享缓存。

示例：

```text
https://reader.curdfu.top:18940
http://localhost:18910
```

这两个地址的阅读端缓存互不共享。

### 缓存大量章节的性能压力

全书缓存可能触发大量请求。

建议：

- 阅读端补齐并发保持 2 左右。
- 定期刷新统计，不要每章全量扫描。
- 允许取消缓存任务。

### 特殊书籍类型

文字章节最适合该方案。

以下类型需要单独确认：

- EPUB
- CBZ/漫画
- 音频
- 返回 iframe URL 的章节

初期建议只保证文字书籍章节。

## 完成标准

功能完成后，应满足：

1. 现有缓存入口不变。
2. 点击缓存后，服务端缓存和阅读端缓存都会增长。
3. 阅读章节时优先使用阅读端缓存。
4. 阅读端未命中时，服务端优先使用本地缓存。
5. 服务端不可用时，已写入阅读端缓存的章节仍可读。
6. 删除服务端缓存和删除浏览器缓存互不影响。
7. 强制刷新可以绕过阅读端缓存并更新内容。
